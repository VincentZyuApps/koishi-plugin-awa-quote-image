# 2026-06-26 给 QQ 官方 Bot 增加 quote 适配

## 背景

Koishi 的 QQ 生态有两个适配器：

| 名称 | 包名 | 简称 | GitHub |
| --- | --- | --- | --- |
| Koishi 官方内置适配器 | `koishi-plugin-adapter-qq` | **内置适配器** | [![](https://img.shields.io/badge/satori-8b5cf6?logo=github)](https://github.com/satorijs/satori/tree/main/adapters/qq) |
| 社区第三方适配器 | `koishi-plugin-adapter-qq-crack` | **crack适配器** | [![](https://img.shields.io/badge/crack-8b5cf6?logo=github)](https://github.com/koishi-shangxue-plugins/koishi-plugin-adapter-qq-crack) |

插件 awa-quote-image 在 OneBot 和 **crack适配器** 上能正常通过 `session.quote` 获取引用消息内容，但**内置适配器**的 `session.quote` 始终为空。需要让插件在内置适配器上也正常工作，且**不修改适配器本身**。

三个核心问题：**获取被引用消息内容**、**获取头像**、**获取用户名**。

---

## 一、获取被引用消息内容

### QQ API 直接写法

官方 QQ Bot API 在群聊/C2C 事件的原始 payload 中通过三个字段传递引用信息：

| 字段 | 说明 | 始终存在? |
|---|---|---|
| `message_scene.ext` | 字符串数组，含 `ref_msg_idx=REFIDX_xxx` 和 `msg_idx=REFIDX_xxx` | ✅ 总是 |
| `message_type` | `103` = 引用消息（MSG_TYPE_QUOTE） | 大部分情况 |
| `msg_elements[0]` | 引用消息时，`[0]` 内联了被引用消息的 `content`、`msg_idx`、`attachments` | ❌ 仅 `message_type===103` 时有 |

**核心发现**：内置适配器的 TypeScript 类型没声明 `msg_elements` 和 `message_type`，但**运行时数据确实有**。直接从原始事件读取即可。

参考实现：`openclaw-qqbot` 的 `parseRefIndices` 函数先扫描 `ext` 数组，再在 `message_type===103` 时用 `msg_elements[0].msg_idx` 覆盖。

```typescript
const d = (session as any).qq?.d
let refMsgIdx, msgIdx
if (d.message_scene?.ext) {
  for (const ext of d.message_scene.ext) {
    if (ext.startsWith('ref_msg_idx=')) refMsgIdx = ext.slice('ref_msg_idx='.length)
    if (ext.startsWith('msg_idx=')) msgIdx = ext.slice('msg_idx='.length)
  }
}
// 缓存命中或用 msg_elements[0]
if (d.message_type === 103 && d.msg_elements?.[0]?.content) {
  content = d.msg_elements[0].content
}
```

**注意**：没有 API 可以按 REFIDX 反查消息内容（`GET /v2/groups/{id}/messages/{id}` 返回 405），所以不能依赖 API，只能靠事件内联数据 + 本地缓存。

### 通用思路

所有平台的引用消息获取遵循**三优先原则**：

```
优先 1: session.quote?.content    ← OneBot / crack适配器
优先 2: 原始事件解析               ← 内置适配器（msg_elements[0]）
优先 3: 缓存命中                   ← 同一会话内之前见过
都拿不到: 报错
```

### Koishi 实现（awa-quote-image）

在 `src/qq.ts` 中，`resolveQQData` 函数实现了这个逻辑：

```typescript
// 1) 从 ext 取 msg_idx / ref_msg_idx
// 2) 缓存当前消息内容（供以后引用）
msgCache.set(msgIdx, { content, userId, username })
// 3a) 缓存命中 → 直接返回内容+作者
if (cached) return { content: cached.content, userId, username, avatar }
// 3b) msg_elements[0] 内联原文
if (d.message_type === 103 && d.msg_elements?.[0]?.content) return { ... }
```

`src/index.ts` 中的 `do_aqt` 调度：

```typescript
if (session.quote?.content) { /* Priority 1 */ }
else if (session.platform === 'qq') { /* Priority 2: resolveQQData */ }
else { /* Priority 3: 报错 */ }
```

## 二、获取头像

### QQ API 直接写法

头像地址格式是固定的公式：

```
https://q.qlogo.cn/qqapp/{botUin}/{userOpenid}/640
```

- `botUin`：机器人 QQ 号（如 `3889704577`），取自 `bot.config.id` 或由用户配置（`qqBotUin`）
- `userOpenid`：用户的 QQ openid（如 `72F2C87032EBF7C3325FB2B2AFD3B75D`），取自 `session.userId` 或 `d.author.id`

内置适配器自己的 `decodeGroupMessage` 使用的就是同一公式：
```typescript
`https://q.qlogo.cn/qqapp/${bot.config.id}/${data.author.id}/640`
```

### 通用思路

头像获取的 fallback 链：

```
优先 1: preUserObj                  ← 插件自己从原始事件拼的
优先 2: session.bot.getUser()      ← 有 getUser 方法的平台
优先 3: session.event.user.avatar  ← 从事件本身拿
```

核心难点：`session.bot.getUser` 不是所有适配器都有。内置适配器就没有。需要做存在性检查：

```typescript
const userObj = typeof session.bot.getUser === 'function'
  ? await session.bot.getUser(userId, channelId)
  : { name: userId, avatar: '' }
```

### Koishi 实现（awa-quote-image）

```typescript
// qq.ts - 拼头像地址
const avatarUrl = (userId: string) => config.qqBotUin
  ? `https://q.qlogo.cn/qqapp/${config.qqBotUin}/${userId}/640`
  : `https://q.qlogo.cn/qqapp/${bot.config?.id}/${userId}/640`

// index.ts - 处理 getUser 不存在
const session_user_obj = preUserObj ?? (
  typeof session.bot.getUser === 'function'
    ? await session.bot.getUser(quoteData.userId, session.channelId)
    : { name: quoteData.userId, avatar: '' }
);
```

同时头像下载失败也不崩：

```typescript
try {
  const avatar_buffer = await ctx.http.file(session_user_obj.avatar)
  avatar_base64 = Buffer.from(avatar_buffer.data).toString('base64')
} catch (e) {
  ctx.logger.warn(`头像下载失败: ${e}`)
}
```

## 三、获取用户名

### QQ API 直接写法

官方 API 的 `UserMessage.author` 类型只声明了 `id: string`，但**运行时实际有 `username` 字段**：

```typescript
d.author.username  // 运行时存在，如 "VincentZyu"
```

直接从原始事件取：

```typescript
const username = d.author?.username
              || session.event?.user?.name
              || session.username
              || session.userId  // 终极兜底
```

### 通用思路

用户名 fallback 链：

```
d.author.username → session.event.user?.name → session.username → session.userId
```

**不因缺少用户名而崩溃**，始终有 fallback。

### Koishi 实现（awa-quote-image）

缓存结构包含 `username`：

```typescript
const msgCache = new Map<string, { content: string; userId: string; username: string }>()
```

存入缓存时记录用户名：

```typescript
msgCache.set(msgIdx, { content, userId: d.author?.id, username: d.author?.username || '' })
```

缓存命中时 fallback：

```typescript
username: cached.username || cached.userId
```

`msg_elements[0]` 路径也从原始事件取：

```typescript
username: d.author?.username || session.event?.user?.name || session.username || session.userId
```

渲染时再 fallback 一次：

```typescript
let usernameArg = session_user_obj?.name || quoteData.userId;
```

## 四、总结

| 问题 | QQ API 直接写法 | 通用思路 | Koishi 实现 |
|---|---|---|---|
| **获取内容** | `msg_elements[0].content` + 缓存 | 三优先原则 | `resolveQQData()` |
| **获取头像** | `q.qlogo.cn/qqapp/{uin}/{uid}/640` | getUser 存在性判断 + fallback | `avatarUrl()` + `preUserObj` |
| **获取用户名** | `d.author.username` | `author → event.user → userId` | 缓存存 username + 渲染兜底 |

### 坑点备忘

1. `session.bot.getUser` 不是所有适配器都有 → 用 `typeof` 判断
2. TS 类型不等于运行时数据 → 内置适配器类型缺字段，但 API 有，用 `as any` 绕开
3. 没有 API 能反查 REFIDX → 必须靠缓存 + msg_elements[0] 内联
4. 头像 URL 必须用 HTTPS → `http://` 会被 `ctx.http.file` 拒绝
5. 内置适配器的 `UserMessage.author` 运行时带 `username`，类型没写但实际有
